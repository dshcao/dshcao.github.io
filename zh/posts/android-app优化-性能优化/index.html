    ANR(Application Not Responding)

    ANR定义

    ANR(Application Not Responding)
    在Android系统中，ANR是四大组件的超时引起的：

    1. Service 超时引起的ANR
      前台Service：SERVICE_TIMEOUT = 20s
      后台Service：SERVICE_BACKGROUND_TIMEOUT = 200s
    2. BroadcastReceiver 超时引起的ANR
      前台广播：BROADCAST_FG_TIMEOUT = 10s
      后台广播：BROADCAST_BG_TIMEOUT = 60s
    3. ContentProvider 超时引起的ANR
      CONTENT_PROVIDER_PUBLISH_TIMEOUT = 10s
    4. Activity界面交互中InputDispatching 输入事件超时引起的ANR
      输入事件包括按键和触摸事件在分发时超过超过5s即判定为ANR，输入事件超时引起的ANR和其它稍有不同，对于输入事件即便执行超过5s只要用户后续没有再产生输入事件就不会产生ANR。

    从上面可以看出ANR也分前台ANR和后台ANR：

    前台ANR用户能感知，比如拥有前台可见的activity的进程，或者拥 有前台通知的fg-service的进程，此时发生ANR对用户体验影响比较 大，需要弹框让用户决定是否退出还是等待
    后台ANR只抓取发生无响应进程的trace，也不会收集CPU信息， 并且会在后台直接杀掉该无响应的进程，不会弹框提示用户

    1. 对于Service当超过一定时间未执行相应操作来触发移除延时消息就会出发ANR
    2. 对于BroadcastReceiver，有序广播的总执行时间超过 2 * receiver个数 * timeout时间 触发ANR，有序广播某一个receiver执行过程超过timeout时长出发ANR。
    3. 对于ContentProvider在进程启动时publish过程可能产生的ANR，系统会直接kill进程并清理相应信息而不会弹出ANR对话框。
    4. 对于Service、BroadcastReceiver,、输入事件发生ANR之后，最终都会调用 AMS.appNotResponding;

    ANR执行流程

    1.发生ANR -> 2.进程接收异常终止信号，开始写入进程ANR信息 -> 3.弹出ANR提示框(不同厂商Rom表现不一)

    user表示用户占用CPU，kernel表示内核占用的CPU，有时可能还有0.2% iowait，iowait表示I/O操作。

    Crash优化

    Crash的原因大概有几种：内存溢出、代码编写问题、WebView问题。

    内存溢出

    导致内存溢出的原因又很多方面，如：

    • 内存抖动
    • 内存泄漏
    • 加载大内存对象 如加载未处理过的大图。

    内存抖动

    频繁创建对象导致GC频繁的一种内存现象。当创建对象过于频繁，GC不及时时可能会导致内存溢出。
    为了防止内存抖动，应注意：

    • 避免循环中创建对象
    • 避免循环调用的方法(onDraw)中创建对象
    • 允许复用的情况下使用对象池进行缓存，比如：Handler的message单链表(obtain)

    内存泄漏

    • 集合问题 使用集合时只有添加没有删除元素，比如EventBus
    • 静态成员/单例 作为GCRoot持有短生命周期对象(如：Activity)导致其无法释放
    • 未关闭释放资源 网络、文件流等未关闭。如FileOutputStream未close
    • 非静态内部类 如Handler postDelay一个匿名Runnable，退出Activity时消息没有处理完。
    • 系统Bug WebView、InputMethodManager
    • 其他 Activity中手动注册广播未调用unregisterReceiver(receiver)， Service执行完未stopSelf()，第三方框架如EventBus未调用unregister(subscriber)解除注册

    SparseArray性能高的原理

    res下图片文件的大小和Android加载到内存成为bitmap对象大小是两码事

    res下图片文件的大小指的是以某种格式压缩后的大小。
    加载到内存成为bitmap的大小只和图片的图片的宽高和格式(ARGB888或RGB565等)有关。
    一个是解码前的文件，一个是解码后的bitmap对象，同样的bitmap对象经过不同算法压缩大小肯定不一样。

    同样的图片，放在drawable-hdpi和放在drawable-xxhdpi下加载到内存是不一样的。如果手机像素密度匹配drawable-xxhdpi，如果drawable-xxhdpi没有相应的图片，Andrid会加载drawable-hdpi中相应的图片，但会相应进行缩放。比如drawable-hdpi图片是ARGB-110x132，而手机匹配的是drawable-xxhdpi，Android会将图片是放大，比如：ARGB_8888-960x720–>ARGB_8888-1920x1440，即加载到内存的bitmap对象所占内存会相应放大，占用更多内存。

    java内存回收算法

    1. 标记清除算法
      标记所有要清除的对象
      同一回收所有被标记的对象
    2. 复制算法
      实现简单运行高效
      浪费一半空间，代价大
    3. 标记整理算法
      标记过程与标记清除算法一样
      存活对象往一端移动
      清理其余内存
      避免了标记清除算法的内存碎片，避免复制算法的空间浪费
    4. 分代收集算法
      结合多种收集算法优势
      新生代对象存活率低，复制
      老年代对象存活率高，标记整理

    Android内存管理机制

    1. 内存弹性分配，分配值与最大值受具体设备影响
    2. OOM场景：1.手机内存不足2.超过分配给app的最大内存
    3. Dalvil仅固定一种回收算法，ART回收算法可运行期选择，如app前台时响应速度更重要，选择简单的标记清除算法，后台时选择标记整理算法
    4. ART具备内存整理能力，减少内存空洞
    5. Low Memory Killer机制 前台进程 可见进程 服务进程 后台进程 空进程

    内存抖动解决：
    定义：内存频繁分配和回收导致内存不稳定
    表现：频繁GC，内存曲线呈锯齿状
    危害：卡顿，OOM。因为频繁创建对象导致内存不足及碎片，不连续的内存碎片导致无法被分配产生OOM
    内存抖动可以通过Memory Profile record录制一端内存查看具体原因

    内存泄漏
    定义：内存中存在已经没有用的对象
    表现：内存抖动，可用内存逐渐减少
    危害：内存不足，频繁GC，OOM

      确实卡顿了，但卡顿堆栈可能不准确
      和OOM一样，最后的堆栈是表象，不是真正问题
      优化：获取监控周期内的多个堆栈而不仅仅是最后一个堆栈，但会导致海量卡顿堆栈处理，高频卡顿上报量太大，服务端有压力。分析：一个卡顿多个堆栈大概率有重复，可以对一个卡顿堆栈进行hash排重，找出重复的堆栈，这样可以极大减少堆栈展示量同时更搞笑找到卡顿堆栈

    Android 线程调度原理剖析

    1. 线程调度原理
      任何时刻只有一个线程占用CPU处于运行状态
      多线程并发 轮流获取CPU使用权
      JVM负责线程调度：按照特定机制分配CPU使用权
      线程调度模型：
      1. 分时调度模型 轮流获取均分CPU时间
      2. 抢占式调度模型：优先级高的采用，JVM采用
    2. Android 线程调度
      1. nice值
        process中定义，值越小优先级越高，默认是Process.THREAD_PRIORITY_DEFAULT 0
      2. cgroup
        更严格的群组调度策略，如：类似后台线程组，前台线程组。保证前台线程获取更多CPU
        线程优先级具有继承性，比如在UI线程中创建的线程具有和UI线程一样的优先级

    HandlerThread
    自带消息循环的线程，串行执行，长时间运行，不断从队列中获取任务
    IntentService
    继承Service内部创建HandlerThread
    异步，不占用主线程
    是个service优先级较高不易被kill

    线程池：
    易复用 减少频繁创建销毁时间
    功能强大 定时 任务队列 并发控制

    提供基础线程池供各个业务使用，避免各个业务维护自己一套线程池，线程太多

    使用AOP监控关键异步任务
